Chapter 01 · The Problem
What problems do small farmers face every day?
Tamil Nadu has over 63 lakh registered small farmers. Most own smartphones. None have access to free, personalised, local farm intelligence.
🌾
No crop guidance
Farmers decide what to plant purely on tradition and intuition. No soil analysis, no weather data, no yield history for their specific district. A wrong crop choice means an entire season — and income — lost.
💰
Market price blindness
Farmers sell at whatever the mandi offers on the day they arrive. No price trend data, no forecasts. They consistently lose 20–35% of potential income by selling at the worst time.
🗺️
Generic advice ignores local reality
Every crop app gives the same advice regardless of where the farmer is. But Erode's red laterite soil and Thanjavur's delta alluvial land are completely different. Advice must be district-specific.
🐄
No animal health support
80% of small farmers own livestock — cows, goats, hens, pigs. When an animal falls sick, the nearest government vet may be 40 km away. Every hour of delay risks losing the animal and the family's income.
🌡️
Late disease detection
Crop disease symptoms are noticed only after 40–60% of the field is already damaged. Weather-based early warning could prevent entire losses — but no free tool exists for this.
💬
Expert access is expensive and far
Private agronomists and vets charge per visit. Government services are understaffed and geographically inaccessible. Small farmers have no reliable, free expert to call.
🎯 Flora's mission in one line
Put expert-level crop intelligence and animal health management in every small farmer's pocket — free, in their language, personalised to their land and district.

Chapter 05 · Crop Intelligence Flow
Crop module — every step from tap to result
From the farmer opening the Soil tab to seeing 5 AI-powered predictions on screen — in under 3 seconds.
1
Farmer opens Soil tab → fills readings
Inline context questions appear first: how many acres this season? which season (Kharif/Rabi/Zaid — auto-detected from the calendar month but farmer can override)? Then the farmer enters N, P, K, pH, and moisture % from their soil test kit. The "Use Mock Data" button auto-fills district-realistic values for quick demo or testing without a kit.
SoilEntryScreenMockSensorServiceplanned_acres stored to soil_scans
2
POST /api/scan/submit → JWT validated → soil_scan saved
Axios sends the payload. The JwtFilter in Spring Boot intercepts the request, validates the token signature and expiry, and extracts the farmer_id from the claims. If the token is invalid, a 401 is returned immediately. If valid, the ScanController delegates to ScanService which saves a SoilScan entity to the soil_scans table — N, P, K, pH, moisture, planned_acres, season, and data_source (MANUAL/MOCK/IOT).
POST /api/scan/submitJwtFiltersoil_scans tableScanService.saveScan()
3
Spring Boot fetches live weather for farmer's district
Using the farmer's district from the farmers table, ScanService calls the OpenWeatherMap API. It retrieves current temperature (°C), humidity (%), rainfall (mm/day), and wind speed, plus the full 7-day forecast as a JSON blob. All data is saved to the weather_snapshots table with a foreign key to scan_id.
OpenWeatherMap free tierweather_snapshots tableforecast_json BLOB column
4
Query district_crop_suitability for local intelligence
Spring Boot queries the district_crop_suitability table for the farmer's district and gets suitability scores (0.0–10.0) for every crop, average yield per acre, and best season. These scores are passed to Flask alongside the soil and weather data so the AI model output can be re-ranked by local agricultural knowledge.
district_crop_suitability table38 TN districts × 22+ cropsTNAU data source
5
RestTemplate calls Flask POST /predict/all — 5 models run
Spring Boot sends one HTTP call to the Flask AI microservice with the full merged payload: soil values, weather snapshot, 7-day forecast, season, district, planned acres, and district suitability scores. Flask has all 5 models pre-loaded in memory at startup (not reloaded per request), so this call takes only ~200–400ms.
RestTemplatePOST /predict/allflora-ai Flask service on EC2models pre-loaded in RAM
6
5 ML models produce output — location score re-ranks crops
Random Forest → top 3 crops with raw confidence %. Location scoring multiplies each crop's confidence by its district suitability score, so Thanjavur farmers see Rice #1 and Erode farmers see Turmeric #1 for identical soil readings. Linear Regression → fertilizer gap (ideal NPK − current NPK) converted to Urea/DAP/MOP kg per acre. Decision Tree → disease risk LOW/MEDIUM/HIGH with disease name and action. Rule+ML hybrid → irrigation: days until next irrigation and volume in mm/acre. LSTM → 21-day price forecast with best sell window.
Random Forest (Model 1)Regression (Model 2)Decision Tree (Model 3)Rule+ML (Model 4)LSTM (Model 5)
7
Prediction saved to MySQL → translated → returned to app
Spring Boot saves the Prediction entity to the predictions table (all 5 model outputs + location yield score, linked to the scan_id). The advisory text is passed through i18n to translate crop names, action steps, and timing advice into the farmer's preferred language (EN/TA/HI). The complete prediction JSON is returned to the React Native app.
predictions tablei18n EN/TA/HIscan_id foreign key
8
PredictionResultScreen renders all 5 outputs
React Native renders scrollable cards: ① Top 3 crops with confidence % + location yield score side by side. ② Fertilizer table — Urea/DAP/MOP kg/acre + application timing. ③ Disease risk badge (green/amber/red) + disease name + action steps. ④ Irrigation card — "Irrigate in N days, apply X mm/acre". ⑤ 21-day price chart with best sell window highlighted in green. All in Tamil if the farmer chose Tamil. Shareable as an image. Saved to scan history.
Total time: ~1.5–3 secondsOffline cached to AsyncStorageShareable image export

Chapter 07 · Expert Connect Flow
Expert Connect — real-time chat from open to resolved
When AI predictions are not enough, a human expert responds in real time. Powered by WebSocket STOMP protocol — the same technology that powers WhatsApp and Slack.
1
Farmer opens Chat tab
Chat Home screen shows all previous conversation threads. Each card shows: category badge (CROP/ANIMAL), current status (OPEN/RESOLVED), assigned expert's name, last message preview, and an unread message count badge. New users see an empty state with a "Start your first chat" prompt.
2
Farmer describes the problem
Flora asks two context questions: crop issue or animal health? Then a brief 1–2 line description. This subject line is saved to chat_sessions.subject and displayed to the expert immediately. The expert sees farmer's district, current crop, and animal types before they read a single message — full context, zero back-and-forth setup.
POST /api/chat/sessionscategory: CROP or ANIMALsubject pre-fills expert panel
3
WebSocket STOMP connection established
React Native connects to the Spring Boot WebSocket endpoint /ws/chat using the STOMP subprotocol. The app subscribes to /topic/chat/{sessionId}. From this moment, any message sent by the expert on the admin panel is pushed to the farmer's phone the instant it's typed — no polling, no manual refresh, no delay.
WebSocket STOMP/ws/chat handshake/topic/chat/{sessionId}
4
Real-time conversation
WhatsApp-style chat UI: farmer messages on the right (green bubbles), expert on the left (white bubbles). Expert's name and designation shown in the conversation header. Online/offline indicator. Delivery ticks: sent (✓), delivered (✓✓), read (✓✓ blue). All messages saved to chat_messages table with is_read flag and sent_at timestamp. Unread badge on Chat tab in bottom navigation updates in real time.
chat_messages tablesender_type: FARMER/EXPERTis_read flag updated on open
5
Session resolved → farmer rates expert
Expert marks session RESOLVED on the admin panel. Farmer is prompted with a star rating screen (1–5 stars, optional free-text comment). Rating saved to chat_sessions.farmer_rating. Status changes to RESOLVED. Session remains viewable in chat history forever — farmer can reference the expert's advice at any time.
PUT /api/chat/sessions/{id}/resolve1–5 star ratingarchived in chat history
🔌 Why WebSocket, not REST polling?
REST polling (calling the API every few seconds to check for new messages) wastes bandwidth, creates visible delay, and drains phone battery. WebSocket keeps a persistent two-way connection open — the server pushes messages to the farmer's phone instantly the moment the expert sends them. This is how WhatsApp and Slack work. Building this in a portfolio project signals backend depth most candidates don't have.
🔗 Seamless integration with Animal Module
When the Symptoms Checker returns vet_visit_required = true, a "Connect to Expert" button appears. Tapping it automatically opens a new chat session with category=ANIMAL and subject pre-filled: "Animal: Lakshmi (HF Cow). Symptoms: Fever, reduced milk, limping. Predicted disease: FMD (78% confidence)." The expert has everything before the farmer types a word.

Chapter 08 · Proactive Intelligence
Daily alerts — Flora speaks to you first
Most apps wait for you to open them. Flora's scheduler proactively checks every farmer's data at their chosen time and alerts only when action is needed.
1
@Scheduled job wakes at each farmer's alert_time
Spring Boot's AlertScheduler runs continuously. It reads all active farmers from the database and checks each one's alert_time column. A 5 AM farmer gets their alerts at exactly 5 AM. A 6 PM farmer gets them at 6 PM. This is fully personalised scheduling — not a generic broadcast to everyone at the same time.
Spring @Scheduledfarmers.alert_time columnper-farmer execution
2
Fresh weather fetched for farmer's district
For each farmer with a recent soil scan, the scheduler fetches today's updated weather from OpenWeatherMap for their district. This fresh weather is compared against the weather from when their last prediction was run — detecting any meaningful changes in humidity, rainfall, or temperature.
3
6 alert checks run in parallel for each farmer
The scheduler evaluates all six alert types simultaneously and only creates an alert record if the condition is genuinely triggered — no noise, no false alarms.
DISEASEDid the disease risk level worsen since the last prediction? Humidity crossed the threshold for the current crop?
IRRIGATIONIs irrigation due today based on the prediction's countdown and today's moisture/rainfall data?
PRICEHas the mandi price crossed the farmer's target threshold? Is the LSTM-predicted best sell window active today?
VACCINATIONIs any registered animal's vaccination due_date within the next 3 days?
MILK DROPHas any animal's yield dropped 20%+ over 3 consecutive days compared to their prior 7-day average?
WEATHERIs extreme weather (heavy rain, drought, frost) forecast in the next 48 hours that affects the current crop?
4
Alert record created → FCM push sent → app badge updates
For each triggered check, one alert row is created in the alerts table with type, severity (LOW/MEDIUM/HIGH), message text (in the farmer's language), is_read=false, and is_pushed=false. Firebase Admin SDK sends a targeted FCM push to the farmer's device_token. The Alerts tab badge in the app increments live.

🔒 Security: BCrypt password hashing
The password is passed through Spring Security's BCryptPasswordEncoder before any database write. BCrypt applies a random salt and 10 rounds of hashing. The original password cannot be recovered. Even if the database is fully compromised, all passwords remain protected.

Chapter 14 · Key Differentiator
Location intelligence — the feature no other app has
The same soil test results in two different Tamil Nadu districts produce different crop recommendations — because district-level agricultural context is baked into every prediction.
How location scoring works
The Random Forest model produces raw confidence percentages based purely on soil + weather data. The location layer multiplies each crop's confidence by the district suitability score from the TNAU table, creating a weighted final score.
RF gives: Rice 78%, Cotton 74%, Maize 68%

Thanjavur (Rice suitability = 9.5)
Rice weighted = 78 × 9.5 = 741 → ranks #1 ✅

Coimbatore (Cotton suitability = 9.1)
Cotton weighted = 74 × 9.1 = 673 → ranks #1 ✅
🗄️ district_crop_suitability — how it's loaded
The table is pre-seeded using a Flyway SQL migration script containing TNAU + TN Agriculture Department data for all 38 districts. It is read-only at runtime and can be updated by running a new Flyway migration when better data becomes available. No retraining of ML models required — location intelligence is a separate, maintainable data layer.

Chapter 15 · System Architecture
Complete data pipeline — how all four layers connect
The exact internal journey of a single crop scan — from the farmer's finger tap to the prediction card on screen.
📱
React Native sends POST /api/scan/submit via Axios
JSON payload: {nitrogen, phosphorus, potassium, ph_level, moisture_percent, planned_acres, season}. HTTP header: Authorization: Bearer <JWT>.
Axios HTTPJSON bodyJWT Authorization header
🔒
Spring Security JwtFilter intercepts → validates → extracts farmer_id
The filter validates the token's signature (HMAC SHA-256), checks the expiry claim, and extracts the farmer_id from the subject claim. If the token is invalid or expired, a 401 Unauthorized response is returned immediately. Valid requests proceed to ScanController.
⚙️
ScanController → ScanService → soil_scans saved to MySQL
ScanController receives the request, delegates to ScanService. A new SoilScan JPA entity is constructed with all soil fields, the farmer_id, and a generated UUID scan_id, then persisted via the ScanRepository (Spring Data JPA).
ScanRepository.save()soil_scans tableHibernate auto-DDL
🌤
WeatherService calls OpenWeatherMap → saves weather_snapshot
ScanService calls WeatherService with the farmer's district. WeatherService makes a synchronous GET to OpenWeatherMap /forecast, parses the JSON response, builds a WeatherSnapshot entity (with the raw forecast blob), and saves it to weather_snapshots linked to the scan_id.
OpenWeatherMap free tierweather_snapshots table7-day forecast JSON stored
📍
SuitabilityService queries district_crop_suitability
A SELECT query fetches all rows where district_name = farmer's district. The result is a Map<String, Double> of crop → suitability score. This map is appended to the Flask payload so location intelligence is applied inside the ML layer itself.
SuitabilityRepository.findByDistrict()district_crop_suitability lookup
🐍
RestTemplate sends merged payload to Flask → 5 models run
One HTTP POST to Flask /predict/all with the full merged payload. Flask has all 5 model files pre-loaded from S3 at startup (joblib.load() and tf.keras.models.load_model()). Models run. Location suitability multiplied into crop rankings. Response returned as JSON in ~200–400ms.
RestTemplate.postForObject()Models pre-loaded at Flask startup~200–400ms Flask response
💾
Prediction saved to MySQL → text translated → JSON returned
PredictionService builds a Prediction entity with all 5 model outputs, links it to the scan_id, and persists to predictions table. Advisory texts (action steps, timing) are passed through the i18n MessageSource for the farmer's language. The complete response is serialised and returned to the app in the HTTP response body.
PredictionRepository.save()i18n MessageSource for EN/TA/HIpredictions table
📱
React Native renders PredictionResultScreen — total time: ~2 seconds
Axios receives the response, React state is updated, PredictionResultScreen unmounts SoilEntryScreen and mounts the result view. All five model output cards are rendered. Data is simultaneously written to AsyncStorage for offline access. Total wall-clock time from farmer's tap to prediction on screen: approximately 1.5–3 seconds.
